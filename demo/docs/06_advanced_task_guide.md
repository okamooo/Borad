# 上級課題：実装ガイド＆ドハマりチェックリスト（【手順7, 8】）

本課題は、PDF資料『おすすめの作成順番』の以下のステップに対応しています。
*   **7. エラーチェック処理(上級課題)**
*   **8. 1回のSQLで記事とコメントを一括で消す(上級課題)**

掲示板開発の総仕上げとして、**「ユーザー体験を損なわない高度なバリデーション制御」**と**「データベースの機能を活かした高度なSQL制御（一括削除）」**を習得することが目的です。

## 1. 実装の重要ポイント

### ① ピンポイントエラー表示（UIとモデルの連動）
*   **課題**: 画面に複数の記事（コメント入力欄）が並んでいる状態で、1つの入力欄でエラーが起きた際、すべての入力欄にエラーメッセージが乱舞してしまう。
*   **解決の思考プロセス**: Thymeleafの `th:errors` は、単純に「Formオブジェクトにエラーがあるか」しか見ません。これを解決するには、**「エラー情報（BindingResult）」と「エラーが発生した親記事ID（articleId）」をセットでModelに保持**し、HTML側で「IDが一致する枠内でのみエラーを表示する」という条件分岐（フィルタリング）を組み込む必要があります。

### ② 1回の指示で親子一括削除（初級の教えとの繋がり）
*   **課題**: 「親（記事）を消す前に子（コメント）を消す」という初級で学んだRDBの鉄則（外部キー制約）を守りつつ、Javaからは「1回のSQL発行」で済ませる。
*   **教育的背景**: 初級では「人間（Java）が順序に責任を持つ」ことを学びました。上級では**「DBやSQLの仕組みに責任を委譲し、Javaのコードをシンプルに保つ」**というアーキテクチャ設計の視点を学びます。「ルールを無視するのではなく、ルールを守る処理をカプセル化（自動化）する」のがテーマです。

## 2. 一括削除を実現する4つのアプローチ

Java（Repository）を変更せずに、1回の呼び出しで親子を削除する方法には、現場の要件に応じていくつかのアプローチがあります。

### アプローチ1：ON DELETE CASCADE（連鎖削除制約）
DBのテーブル定義自体を変更し、「親が消えたら子も道連れにする」設定を付与します。
*   **メリット**: Java側は親を消すコード1行で済む。
*   **デメリット**: 「勝手に消える」ため、意図せぬ大量データ喪失のリスクがあり、実務では使用が禁止される現場もあります。

### アプローチ2：マルチステートメント（セミコロン連結）
`DELETE FROM comments WHERE... ; DELETE FROM articles WHERE...` のように、2つのSQLをセミコロンで繋げて1回で送信します。
*   **教育的メリット**: 「まず子、次に親」という初級で学んだ順序がコードに明記されるため、受講生が最も納得しやすい方法です。

### アプローチ3：CTE（共通テーブル式 / WITH句）
```sql
WITH deleted_comments AS (DELETE FROM comments WHERE article_id = :id) 
DELETE FROM articles WHERE id = :id;
```
*   **メリット**: DBの設定を変更せず、1つのアトミック（不可分）な処理として安全に実行できます。

### アプローチ4：ストアドファンクション（PL/pgSQL）
DB側に「子を消して親を消す関数」を定義しておき、Javaからは `SELECT delete_article_and_comments(:id)` と1回呼び出します。
*   **メリット**: 複雑な処理をDB側に完全に隠蔽（カプセル化）できます。

## 3. よくある詰まりどころ（トラブルシューティング）

### 🚨 バリデーションエラーが消えない / 全部に表示される
*   **症状**: 1箇所で入力ミスをした後、正しい値を入力してもエラーメッセージが残り続ける、または無関係な記事に入力してもエラーが出る。
*   **原因**: エラー時に `Redirect` ではなく `Forward`（メソッドの直接呼び出し）で画面を再描画しているため、Modelに古い `BindingResult` が残存している。または、`articleId` による絞り込み判定が抜けている。
*   **対策**: Controllerの引数で `@Validated` を付与した Form の直後に必ず `BindingResult` を配置し、エラー発生時のフローを独立させてください。

### 🚨 「ON DELETE CASCADE」を設定したのに外部キー制約エラーが出る
*   **症状**: ALTER TABLE文を実行したのに、親を削除しようとすると依然としてエラーになる。
*   **原因**: 既存の外部キー制約が残ったまま、名前の違う新しい制約を追加してしまっている。
*   **対策**: 必ず `DROP CONSTRAINT` で既存の制約を削除してから、CASCADE付きの制約を `ADD CONSTRAINT` してください。

### 🚨 （マルチステートメント等で）SQLの構文エラー
*   **症状**: SQLを複数繋げたら `BadSqlGrammarException` が発生する。
*   **対策**: Javaの文字列結合時のスペース忘れ（例：`WHERE id = :idDELETE FROM` になってしまう）に注意し、前後のスペースを意識してください。
